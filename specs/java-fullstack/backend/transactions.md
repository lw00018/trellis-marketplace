# 事务规范

## 事务边界

- 事务必须放在 Service 层公共方法上。
- Controller、Mapper、Entity 不承担事务边界。
- 一个事务只覆盖必须保持一致的数据库写入，不包裹远程调用和长时间计算。
- 读操作使用 `readOnly = true`，写操作显式声明超时时间。

## 自调用陷阱

Spring AOP 代理不会拦截同类内部方法调用。以下情况事务不会按预期生效：

```text
public method A -> this.methodB()
methodB 上的 @Transactional 不会通过代理触发
```

解决方案：

- 把事务方法放到另一个 Service。
- 从代理对象调用事务方法。
- 重新设计事务边界，让外层公共方法承担事务。

## 传播行为

| 传播行为 | 使用场景 |
| --- | --- |
| REQUIRED | 默认选择，加入现有事务或创建新事务 |
| REQUIRES_NEW | 独立审计日志、补偿记录等必须单独提交的操作 |
| SUPPORTS | 可参与事务也可非事务执行的只读查询 |
| NEVER / NOT_SUPPORTED | 明确禁止事务的长耗时任务 |

不要为了“保险”滥用 `REQUIRES_NEW`，它会破坏调用方原子性预期。

## 回滚规则

- 默认只对 RuntimeException 和 Error 回滚。
- 需要对受检异常回滚时必须显式配置 `rollbackFor`。
- 捕获异常后如果不重新抛出，事务不会自动回滚。
- 禁止吞掉异常后返回成功。

## 外部系统

- 事务内禁止执行不可控远程调用。
- 支付、消息、第三方接口必须设计幂等、重试和补偿。
- 数据库事务不能保证跨系统一致性，跨系统流程使用最终一致性方案。

## 检查清单

- 是否所有写操作都位于明确事务边界内。
- 是否存在同类自调用导致事务失效。
- 是否在事务内做了远程调用或长时间操作。
- 是否明确异常回滚规则。
- 是否有并发更新和幂等保护。
