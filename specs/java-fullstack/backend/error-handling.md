# 错误处理规范

## 异常分层

| 类型 | 说明 |
| --- | --- |
| 参数异常 | 请求格式、必填、范围不合法 |
| 业务异常 | 用户可理解的业务规则失败 |
| 权限异常 | 未认证、无权限、资源不归属 |
| 外部依赖异常 | 数据库、缓存、第三方服务失败 |
| 系统异常 | 未预期服务端错误 |

## 统一异常处理

- 必须使用全局异常处理器统一转换响应。
- 不得在每个 Controller 中重复 try-catch。
- 未预期异常返回通用错误信息，并记录完整日志。
- 业务异常必须携带稳定错误码。

## 错误码

- 错误码必须稳定，不随文案变化。
- 错误码使用大写下划线，例如 `USER_NOT_FOUND`。
- 错误码必须按领域归类，避免所有错误都叫 `FAIL`。
- 公开 API 的错误码变更属于契约变更。

## 捕获异常

- 禁止捕获异常后不处理。
- 禁止用异常控制正常业务流程。
- 捕获异常后必须记录上下文或转换为更有语义的异常。
- 需要回滚事务时不要吞掉异常。

## 日志关联

- 异常日志必须包含 traceId、用户标识、业务键和关键上下文。
- 对外响应包含 traceId，方便用户反馈后定位日志。
- 日志中不得输出敏感字段。
