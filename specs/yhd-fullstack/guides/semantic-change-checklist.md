# 语义变更检查清单

本清单用于修改字段含义、枚举、状态码、默认值、路由参数或接口数据解释方式时，确保所有依赖点同步更新。

## 什么是语义变更

语义变更不是简单新增字段，而是改变已有数据或状态的解释方式，例如：

- 字段从"是否启用"变为"启用状态枚举"。
- 状态码 `1` 的含义改变。
- 时间字段从秒变为毫秒，或从字符串变为时间戳。
- `null`、`undefined`、空字符串、空数组的含义改变。
- 路由 query 参数从单值变成多值。
- 表单默认值或筛选默认值代表的业务含义改变。
- 权限、菜单展示或按钮可见规则改变。

## 必查位置

### 1. API 与 Model

- [ ] `src/api/` 的入参、响应、上传 URL 或分页参数是否需要调整？
- [ ] `src/model/` 中的类型、枚举、可选字段、联合类型是否同步更新？
- [ ] 接口 mock、测试数据或 fixture 是否仍使用旧语义？
- [ ] 是否需要兼容后端暂未同步的返回值？如果需要，应在系统边界处理，不要散落页面。

### 2. 表单

- [ ] 新增、编辑、详情回填的默认值是否一致？
- [ ] 校验规则是否符合新语义？
- [ ] Select、Radio、Checkbox、Switch 的 value 类型是否正确？
- [ ] 提交前转换是否仍然正确？
- [ ] 重置表单是否会恢复到正确业务默认值？

### 3. 表格、筛选与展示

- [ ] 表格列、状态颜色、Tag 文案、排序规则是否同步？
- [ ] 搜索筛选项和默认筛选条件是否同步？
- [ ] 详情页、列表页、弹窗、导出或复制内容是否还有旧文案？
- [ ] 空状态、异常状态、禁用态是否需要新增分支？

### 4. Store、缓存与路由

- [ ] Pinia Store 中的状态、getter、setter 是否仍使用旧语义？
- [ ] `localCache` / `sessionCache` 中是否存在旧值，需要迁移或清理？
- [ ] route params/query 是否需要解析或序列化新格式？
- [ ] keepAlive、标签页刷新、关闭后重开是否会读到旧状态？

### 5. 组件与 Composable

- [ ] 组件 props / emits / slots / expose 类型是否同步？
- [ ] 组件 `config.ts` 中的枚举、状态映射、默认表单值、表格列是否同步？
- [ ] 组件 `composable/` 中的派生状态、watch、提交逻辑是否仍符合新语义？
- [ ] 私有子组件是否通过 props 继续传递旧字段或旧状态？

### 6. 权限、菜单与文案

- [ ] 路由 meta、菜单显示、按钮权限是否需要更新？
- [ ] 页面标题、按钮、placeholder、表格列名、消息提示是否同步？
- [ ] 如果启用 i18n，所有用户可见文案是否统一接入？
- [ ] 是否有硬编码旧文案需要替换？

### 7. 测试

- [ ] Jest 单元测试是否覆盖新语义？
- [ ] 组件测试是否更新 props、交互和渲染断言？
- [ ] 是否需要新增旧值、空值、异常值测试？
- [ ] 是否删除或更新了会误导后续维护者的旧测试数据？

## 搜索建议

变更字段或枚举前先全局搜索：

```bash
rg "字段名|枚举名|旧文案|旧状态码" src tests
```

不要只搜索类型名，也要搜索：

- 接口字段名
- 表格列 dataIndex
- 表单字段 name
- route query key
- 缓存 key
- 文案和状态颜色
- 测试断言和 mock 数据

## 完成标准

- [ ] 新旧语义边界清楚，系统边界处理集中。
- [ ] API、Model、UI、Store、缓存、路由、测试都已同步检查。
- [ ] 没有页面局部保留旧状态映射或旧默认值。
- [ ] 没有为了兼容旧语义引入长期无主的 fallback。
- [ ] 用户可见行为与新语义一致。
